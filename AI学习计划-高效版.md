### Day2（5/14）分支操作

**任务（1小时）：**

1. 打开昨天的 `git-test` 文件夹（20分钟）
   ```bash
   cd ~/git-test        # 回到测试文件夹（如果关了终端）
   git branch            # 看当前有哪些分支
   git branch dev        # 新建一个叫dev的分支
   git checkout dev      # 切换到dev分支
   ```
   然后改一下 test.txt 内容（随便写点新东西），add + commit。再切回 master 分支，看看文件内容有啥变化。

2. 理解为什么要分支（20分钟）

   分支就是"平行世界"。你可以在 dev 分支上随便实验，master 分支不受影响。实验成功了就合并（merge），失败了直接删掉分支，master 毫发无伤。

   这就解决了一个问题：你想改网页但怕改坏——开个分支改，改好了合并，改坏了删掉重来。

3. 练习合并（20分钟）
   ```bash
   git checkout master       # 回到master
   git merge dev             # 把dev的改动合并进来
   git log --oneline --graph # 看图，理解分支历史
   ```

**产出：** 会用分支保护主代码不受影响。

**今天的关键理解：**
> master 分支 = 稳定版（展示给外人看的）
> dev 分支 = 实验版（你瞎折腾的地方）
> 两个世界互不干扰，merge 的时候才合到一起


### Day3（5/15）GitHub——把你的代码推上云端

**前置准备：** 打开 [github.com](https://github.com)，注册账号。用户名建议用跟业务相关的（以后展示给别人看的）。

**任务（1小时）：**

1. 在GitHub网页上创建一个新仓库（15分钟）
   - 点右上角 + 号 → New repository
   - Repository name 填 `git-test`
   - 勾选 Public（公开）
   - 不要勾选任何初始化选项
   - 点 Create repository
   - 页面会跳转，显示一堆命令——把 `git remote add origin...` 那三行命令复制下来

2. 把本地仓库推上去（25分钟）
   ```bash
   cd ~/git-test
   git remote add origin https://github.com/你的用户名/git-test.git
   git branch -M main           # 把master改名叫main
   git push -u origin main      # 推上去！
   ```
   如果提示登录，按浏览器弹窗授权就行。

3. 刷新GitHub页面，看到你的文件了吗？（10分钟）
   - 点一下 `test.txt` 看看
   - 点一下那个提交 hash（一串乱码），看提交详情
   - 点 Insights → Network，看提交图

4. 改点东西再推一次（10分钟）
   ```bash
   echo "第二次修改" >> test.txt
   git add test.txt
   git commit -m "第二次提交"
   git push
   ```
   刷新GitHub，看提交记录又多了一条。

**产出：** 你的代码存在了云端，任何人可以通过链接看到。

**今天的关键理解：**
> GitHub = 你代码的"云盘+朋友圈"
> 本地改了 → git push → 云端同步
> 别人改了 → git pull → 拉到你电脑
> 这就是"远程协作"的基础

### Day9（5/21）Prompt实战①——让AI生成你想要的，不是它想说的

**任务（1小时）：**

今天就是不断问 Claude，不断优化你的问法。核心技巧就四个：

1. **给角色**（15分钟）

   对比这两个 Prompt：
   ```
   A: "写一段产品介绍"
   B: "你是亚马逊顶级卖家。给你的智能水杯写一段产品介绍，面向25-35岁注重健康的美国白领。用英文。"
   ```
   明显B的回答好得多。这就是角色设定的力量。练习：让 AI 扮演"Shopify 资深运营"，帮你分析一个产品卖点。

2. **给约束**（15分钟）
   ```
   "用英文写5个产品标题，每个不超过80个字符，包含关键词'portable'，不要用形容词最高级（best/amazing等）"
   ```
   约束越具体，结果越可控。练习：给自己想象中的产品写一段文案，加3个约束条件。

3. **给示例**（15分钟）
   ```
   "参考这个风格写：[贴一个你喜欢的竞品文案]。写一个类似风格的，产品是[你的产品]"
   ```
   Few-shot prompting 是让 AI 学会你的"品味"最好的方法。

4. **分步骤**（15分钟）
   ```
   "我的产品是[XX]。请按以下步骤帮我：
   第一步：分析目标客户画像
   第二步：提取3个核心卖点
   第三步：基于卖点写一段英文产品描述
   第四步：给出3个SEO优化建议"
   ```
   复杂任务拆成步骤，每一步都可控。

**产出：** 4个 Prompt 模板——角色扮演 / 加约束 / 给示例 / 分步推理。

### Day18（5/30）Agent概念——用一个下午讲清楚

**任务（1小时）：**

今天不动手，理解概念。但不用看视频——我直接讲。

1. **什么是AI Agent**（20分钟）

   普通AI = 你问它答，一次性的
   AI Agent = 它有"手"和"脑"，能自己规划步骤、调用工具、完成任务

   类比：
   - 普通LLM = 一个很聪明的实习生，你问什么他答什么，但不主动做事
   - AI Agent = 同一个人，但给了他电脑权限和任务清单，他能自己完成一个目标

   Agent 的四件套：
   | 组件 | 干什么 | 你见到过 |
   |------|-------|---------|
   | LLM（大脑） | 理解任务、做决策 | 我（Claude） |
   | Tools（手） | 执行具体操作 | Bash、Edit、Read |
   | Memory（记忆） | 记住上下文和偏好 | Memory系统 |
   | Planning（规划） | 把大任务拆成小步骤 | 我每次做事前先分析 |

2. **你已经在用Agent了**（20分钟）

   我就是Agent。你观察我这几次帮你做事时干了什么：
   - 你说"帮我改代码" → 我先读文件，定位位置，然后修改
   - 你说"看看项目结构" → 我 ls 看目录，再逐个读文件
   - 你说"找模板" → 我分析需求，搜索，推荐

   这就是 Agent 的标准工作流：**感知→推理→规划→行动→反馈**

3. **Agent和你的关系**（20分钟）

   三个月后你跟Agent的关系：
   - 你是老板，Agent是你的执行助手
   - 你说"帮我出一份这周的竞品分析报告"
   - Agent自己：搜竞品→读评价→整理成表格→生成报告→保存文件
   - 你只需要审核和微调

**产出：** 理解了 Agent 是什么，以及你现在用的 Claude Code 本身就是Agent。

### Day22（6/3）什么是API——让程序之间说话

**任务（1小时）：**

1. 理解 API 概念（15分钟）

   API = 程序的"外卖菜单"。你不需要进厨房（打开那个程序），看菜单点菜（调API），菜就送到你面前。

   - 你在浏览器输入网址 = 你在调用网站的API
   - 微信扫码支付 = 微信API被调用
   - Claude Code 帮你执行命令 = 调用系统API

2. 亲手调一个公开API（30分钟）

   浏览器地址栏输入：
   ```
   https://api.github.com/users/你的GitHub用户名
   ```
   会显示一堆JSON数据——这就是GitHub API返回的你的用户信息。

3. 理解看到的东西（15分钟）

   ```json
   {
     "login": "你的用户名",
     "public_repos": 3,
     "followers": 0,
     ...
   }
   ```
   这就是 JSON——所有 API 的标准语言。花括号包对象，方括号包列表。

**产出：** 理解了 API 就是"通过网址获取数据"，JSON 就是数据格式。

### Day27（6/8）学看报错 + 调bug

**任务（1小时）：**

编程最核心的技能不是写代码，是**出了错知道怎么办**。

1. 故意犯错练习（20分钟）
   - 在脚本里删一个引号，跑一下，看报什么错
   - 把文件名写错，看报什么错
   - 把API Key删掉，看报什么错
   
   每看到一个错误信息，复制到 Claude Code：
   > "这个报错是什么意思？怎么修？"

2. 学会"遇错四步法"（10分钟）
   ```
   第1步：看报错最后一行（通常告诉你在哪错了）
   第2步：检查那一行的拼写和符号
   第3步：复制报错信息，问 Claude Code
   第4步：修一处跑一次，不要一次改多处
   ```

3. 看 Claude Code 怎么调试（30分钟）
   给 Claude Code 一个故意有bug的脚本：
   > "帮我看看这个脚本为什么跑不过，修好它"

**产出：** 不再害怕报错，知道怎么解决。
